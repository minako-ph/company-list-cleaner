## 1. 背景・目的・成功基準

Googleスプレッドシート上の取引先リスト（社名・法人番号）に対し、表記正規化・法人番号付与・基本情報・**適格請求書（インボイス）発行事業者の登録状況**・補助金/調達実績を一括付与するエディタアドオン。経理・営業の「取引先リスト整備」という制度由来の恒常ペインを、シートから出ずに解決する。

**成功基準（ビジネスKPI、§10）**: 公開後90日でインストール500・有料10人／6ヶ月でインストール2,000・有料転換1%／12ヶ月でMRR¥30,000以上。月10万円ライン＝有料約68人（¥1,480）。

## 2. スコープ／非スコープ／前提

**スコープ**: Google Sheets用エディタアドオン（サイドバーUI）＋課金・照会用バックエンド＋公開に必要なLP・プライバシーポリシー・利用規約。

**非スコープ**: Excel版／スタンドアロンWebアプリ／AI Import（URL→LLM構造化。競合過密のため第1弾テレメトリ確認まで凍結）／Gmail・Drive等シート外への一切のアクセス／レポート生成・ダッシュボード／英語圏向け本格展開（リスティングの英語併記まで）。

**前提（公開ゲート）**:

| ゲート | 状態（2026-07-07） | 影響 |
|---|---|---|
| 国税庁アプリケーションID（インボイス＋法人番号共通） | 申請済み・審査1〜1.5ヶ月 | FR-5の有効化条件。**§9の縮退公開計画あり** |
| Google OAuth審査（sensitiveスコープ） | 未着手（ブランド確認2〜3営業日＋審査1〜3週） | 公開の必須条件 |
| Marketplace掲載審査 | 未着手 | 同上 |
| gBizINFO v2トークン | 取得済み | FR-4/FR-6のデータ源（v1 APIは2026年9月終了、v2実装必須） |

## 3. 利用者

一次: 国内中小企業の経理・営業・バックオフィス担当（法人）。二次: 個人事業主。
**国税庁申請書との整合**: 申請書2.4で利用者区分を「顧客＝一般（法人）・一般（個人）」と届出済み。**税理士を明示ターゲットにしない**（税理士区分は未届出。需要顕在化時は変更手続を検討＝§12未決）。

## 4. 機能要件（FR）

- **FR-1 列マッピングUI**: サイドバーで対象範囲と列の役割（社名列／法人番号列／登録番号列）を指定できる。ヘッダー自動推定＋手動修正。
- **FR-2 表記正規化**: 社名の表記ゆれを正規化する（全角半角、（株）⇄株式会社、前株/後株、余分な空白・法人格の統一）。ルールベースのみ（**v1でLLM不使用**）。
- **FR-3 法人番号解決**: 社名から法人番号Web-API（名称検索）で法人番号を解決する。一意に定まらない場合は候補リスト（名称・所在地）を提示しユーザーが選択。全結果に`confidence`（exact／selected／ambiguous／not_found）を付す。
- **FR-4 基本情報付与**: 法人番号をキーに、所在地・法人種別（法人番号API基本3情報）＋業種・設立・従業員数等（gBizINFO v2）を選択式で付与する。
- **FR-5 インボイス登録確認**〔機能フラグ `INVOICE_ENABLED`〕: **登録番号（法人はT＋13桁法人番号）を指定した照会のみ**で、登録の有無・登録年月日・取消/失効の状況を付与する。§5のコンプライアンス要件に完全従属。国税庁ID到着まで無効。
- **FR-6 補助金・調達フラグ**: gBizINFOから補助金受給・国等との調達実績の有無（および直近件数）を付与する（任意列）。
- **FR-7 結果書き込み**: 結果は**新規列として追記**し、既存セルを上書きしない。行ごとのステータス列（成功／候補選択待ち／エラー理由）を必ず出力する。
- **FR-8 部分失敗と再実行**: 1行の失敗で全体を止めない。「未処理・エラー行のみ再実行」ができる。
- **FR-9 無料枠と使用量表示**: 無料枠は**月50行**。サイドバーに当月使用量と残数を常時表示する。
- **FR-10 ライセンス**: 有料化はライセンスキー方式。サイドバーからStripe Checkoutへ誘導→決済完了メールでキー発行→キー入力で解錠。キーは端末でなくGoogleアカウントに紐づく。解約はStripeカスタマーポータル。
- **FR-11 個人事業主の取扱い明示**: 個人事業主の登録番号は法人番号から導出できないため、**T番号が既にリストにある場合のみ照会可能**である旨をUI・ヘルプ・リスティングに明記する（正直明記）。
- **FR-12 ヘルプ・出典表記**: サイドバー内ヘルプに使い方・制約・§5の出典指定文言を常設する。

## 5. コンプライアンス要件（CR）— 国税庁申請書の記載を要件化。**違反は承認取消リスク＝最優先制約**

- **CR-1**: インボイスWeb-APIへの照会キーは登録番号のみとする（法人はT＋13桁法人番号を機械生成）。
- **CR-2**: 氏名・名称・所在地等、**登録番号以外の情報から公表情報を検索する機能を実装しない**（社名→法人番号の解決は法人番号システム側APIで行い、インボイスAPIには常に登録番号で照会する、という届出済みの適法チェーンのみ許可）。
- **CR-3**: 取得した公表情報を**申請者側サーバ・DBに保存しない**（都度取得・都度破棄）。書き込み先は利用者本人のスプレッドシートのみ。
- **CR-4**: 公表サイトの全件ダウンロードファイルを利用しない（申請書2.3＝無し）。
- **CR-5**: アクセスログは「利用者ID・取得日時・照会した登録番号」の3点に限定する（申請書2.6.2どおり。社名・照会結果はログに残さない）。
- **CR-6**: 出典指定文言を表示する。国税庁（インボイス）:「このサービスは、国税庁適格請求書発行事業者公表システムのWeb-API機能を利用して取得した情報をもとに作成しているが、サービスの内容は国税庁によって保証されたものではない」／国税庁（法人番号）: 法人番号版の同型文言／gBizINFO:「出典：経済産業省 Gビズインフォ」。
- **CR-7**: OAuthスコープは `spreadsheets.currentonly`・`script.external_request`・`script.container.ui` の3点に固定する。**restrictedスコープ（Gmail/Drive全体等）は恒久的に不使用**（CASA Tier2監査＝年$500〜4,500の回避設計。機能要望があっても追加しない）。

## 6. 非機能要件（N）

| # | 要件 |
|---|---|
| N-1 | 公的APIへのリクエストは申請者管理バックエンドから直列・適切な間隔（既定1req/秒、インボイスAPI仕様確認後に調整）で送信する（申請書の記載どおり） |
| N-2 | GASの6分実行制限を回避するバッチ設計（サイドバー駆動で50行単位の分割呼び出し・進捗表示・中断再開） |
| N-3 | シートから外部に送るデータは対象列の値（社名・番号）のみ。それ以外のシート内容を読み取らない・送信しない |
| N-4 | 監視: バックエンドのエラー率・公的APIの障害を検知しオーナーに通知。障害時はサイドバーに状態表示（無応答にしない） |
| N-5 | 保守は週合計の枠内で月2時間以内を目標。FAQ・アプリ内ヘルプで問い合わせを前倒しで削減 |
| N-6 | UIは日本語を正とし、Marketplaceリスティングは日英併記 |
| N-7 | 可用性の縮退: いずれかの公的APIが停止しても他機能は動作継続（機能単位の分離） |

## 7. 課金要件

| プラン | 価格 | 内容 |
|---|---|---|
| Free | ¥0 | 月50行まで。全機能利用可（量の制限のみ） |
| Pro | **¥1,480/月（税込）** | 月10,000行までのフェアユース上限（公的APIへの礼節＝N-1保護。超過は個別相談と明記） |

決済はStripe（Checkout＋カスタマーポータル）。年額プランはv1.1判断（§12）。無料枠カウントは利用量データでありCR-3（公表情報の非保存）に抵触しない。

## 8. 受入基準（公開条件）

1. FR-1〜12・CR-1〜7・N-1〜7を満たす（**CRはコードレビューでの明示的検証項目**とする）。
2. OAuth検証（ブランド＋sensitive審査）通過。スコープが§CR-7の3点のみであることを提出物で確認。
3. Marketplace掲載審査通過。プライバシーポリシー・利用規約・LPが公開済み。
4. E2E: 新規ユーザーが「インストール→サンプル10行で正規化＋法人番号付与」を3分以内に完了できる。
5. 課金E2E: Checkout→キー発行→解錠→解約が通しで動作。
6. インボイス機能はID到着・検証環境での照合確認後にのみ有効化（§9）。

## 11. リスク

国税庁の不承認・承認遅延（→§9縮退計画）／承認条件違反によるID取消（→CRをコードレビュー必須項目化）／Google審査・ポリシー変更、CASA適用範囲拡大（→CR-7のスコープ最小主義）／gBizINFO v2移行期の仕様流動／公的API障害（→N-7縮退）／**運営者自身の適格請求書発行**: 有料顧客（事業者）から適格請求書を求められる可能性→自身のインボイス発行事業者登録の要否を検討（§12）。

## 12. 未決事項

1. インボイスWeb-API仕様書の確認: 1リクエストあたりの登録番号最大指定件数・提供時間帯・レート（実装初日、公開仕様書で確認）。
2. gBizINFO v2の正式仕様（柱2未決#1と共通。柱2側の確認結果を流用）。
3. Marketplace掲載の最新要件（スクリーンショット規格・審査観点）を提出前に公式docsで確認。
4. 年額プラン（¥14,800/年）の導入判断（v1.1）。
5. 税理士向け訴求の解禁: 需要顕在化時に国税庁への変更手続を確認の上で判断。
6. 運営者自身のインボイス発行事業者登録の要否（顧客からの適格請求書要求への対応方針）。
