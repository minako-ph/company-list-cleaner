## R3-1【重大】ユーザー安定キー方式の確定（無料枠カウントの基盤）

- 対象: 引継書§6「ハッシュ化した安定ユーザーキー（詳細は実装初日に検証）」・§12-5
- 修正: 検証待ちにせず方式を確定する。`Session.getActiveUser().getEmail()`は現行の3スコープ構成では**空文字になるケースがあり信頼できない**（メール取得には追加スコープが必要になり、スコープ3点固定＝CR-7と衝突する）。よって:
  1. **初回起動時にUUIDを生成し`PropertiesService.getUserProperties()`に保存**、以後これを安定ユーザーキーとする（ユーザー毎・スクリプト毎に分離され、追加スコープ不要）。
  2. 無料枠カウント（月50行）はこのUUIDをキーにbackend（Firestore）で管理。
  3. ユーザーがプロパティを消去すれば無料枠がリセットされる悪用余地は**許容**する（上限50行のため実害僅少。対策コードを書かない——複雑化のコストの方が大きい）。
  4. ライセンス紐付けは従来どおり「キー⇄Stripe購読」が真実源（UUIDはあくまで無料枠用）。
- 引継書§12-5（実装初日の検証タスク）は「UserProperties方式の動作確認」に読み替える。

## R3-2【重大】ライセンスキー配布方式: メール送付 → 成功ページ表示＋再表示エンドポイント

- 対象: 引継書§7.3「キーを生成しメール送付」／要件書FR-10「決済完了メールでキー発行」
- 修正: メール送付を一次手段にしない（送信ドメイン設定・SPF/DKIM・送信基盤という**隠れた依存と初期コスト**が発生するため）。確定フローは:
  1. Stripe Checkoutの成功リダイレクト先を `LP/thanks?session_id={CHECKOUT_SESSION_ID}` に設定。
  2. thanksページのJSがbackend `/license/claim` にsession_idを渡す→backendがStripeでsessionを検証→**ライセンスキーをその場で画面表示**（コピー導線付き）。
  3. **再表示手段**として、LPに「キーを忘れた方」フォーム（メールアドレス→backendがStripe顧客を照合→キー再表示）を用意。
  4. メール送信（Resend等の無料枠）は任意の強化であり、v1の必須要件から外す。
- 要件書FR-10・受入基準5のE2Eは「Checkout→成功ページでキー表示→解錠→再表示→解約」に読み替える。
